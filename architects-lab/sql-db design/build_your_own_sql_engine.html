0. Why a SQL engine? category

A: "Don't say 'we need a database.' Everyone says that. What problem does a SQL engine solve that a simple key‑value store doesn't? Be concrete."
Before revealing, decide: what's the one query pattern that forces a SQL engine? Name a concrete failure of a key‑value store for a mutual fund order flow.
A
"Key‑value gives you O(1) by key. Why can't you just use that for everything?"
B
"Because orders need filtering: 'get all IPO allotments with status='PENDING' and quantity > 100'. That's a scan over all keys — terrible when you have 80+ microservices and millions of orders."
A
"So you want secondary indexes? Fine. But that's not a SQL engine yet."
B
"Right. The SQL engine is about *declarative queries*: the user says what, not how. We need to translate that to a plan, then execute it efficiently. That's the whole point."
┌─────────────┐ ┌──────────────┐ ┌─────────────┐ │ SQL query │ ──→ │ parser + │ ──→ │ execution │ │ (declar.) │ │ planner │ │ (iterators)│ └─────────────┘ └──────────────┘ └─────────────┘
🔹 The engine exists because users need rich filtering + joins + aggregations — not just point lookups. Every later stage is about making that declarative promise fast.

1. Parsing: from text to AST parser

A: "Parsing is textbook. But if you write a recursive‑descent parser, you'll hit a problem with the very first 'SELECT'. What's the actual issue?"
Think: what's the hard part about parsing SQL? It's not grammar — it's the context (table names, aliases). Name the failure scenario.
A
"You can generate a parser from yacc. What's the challenge?"
B
"SQL is context‑sensitive: 'SELECT * FROM orders o JOIN users u ON o.user_id = u.id'. The parser needs to resolve 'o' and 'u' as table aliases — but that requires catalog metadata. Without it, you can't validate joins."
A
"So you can't just parse and be done. You need a two‑pass: parse → bind → resolve."
B
"Exactly. First pass builds an AST with raw identifiers. Second pass (binding) resolves them against the catalog. That's where we catch 'unknown column'."
SELECT o.amt, u.name FROM orders o JOIN users u ON o.uid = u.id [AST] → [Binder] → [Resolved AST with table OIDs & column IDs]
🔹 Parsing is only half the story: binding forces a catalog dependency early. That catalog will become central for every later stage.

2. Execution model: iterator or push? exec

A: "You have a resolved AST. How do you actually run it? If you say 'volcano iterator', show me the problem with that model in a FinTech order flow."
Volcano is classic. But what breaks when you have 10 million orders and a 5‑way join? Name the bottleneck.
A
"Iterator model: each operator calls Next() on child. Simple. What's wrong?"
B
"It's pull‑based. If you have a hash join probe, it blocks until the build side is fully materialized. That's fine for small tables, but for NSDL demat holdings (millions of rows), you need memory control."
A
"So you want push? That's harder to compose."
B
"We use a hybrid: pull‑based with memory‑aware operators. Each operator can spill to disk if the build side exceeds a threshold. That way we avoid OOM in the middle of an IPO allotment query."
┌───────┐ Next() ┌────────┐ Next() ┌────────┐ │ Scan │ ──────→ │ Filter │ ──────→ │ Join │ └───────┘ └────────┘ └────────┘ (pull) (pull) (pull + spilling)
🔹 Execution model isn't academic: it dictates memory management and spill logic. Every later operator must respect this.

3. Storage: heap vs. columnar storage

A: "You need to store rows. Heap file is simple. But with 80 microservices and PolicyBazaar insurance journeys, you have wide tables. What fails?"
Why not just store rows as JSON? Point to the exact query that makes columnar storage necessary.
A
"Heap file: append rows. Good enough?"
B
"For OLTP, yes. But our mutual fund order flow needs analytics: 'average NAV per AMC over last quarter'. That's a full scan of a wide row (200+ columns) — we'd read tons of unused data."
A
"So you want columnar? That's a trade‑off: writes become slower."
B
"We go hybrid: hot rows (recent orders) stored row‑oriented for fast updates; historical data stored columnar for analytics. The engine routes queries based on the time range."
Row-store: [ id | amt | nav | timestamp | ... ] (wide) Columnar: [ id, id, id, ... ] [ amt, amt, ... ] [ nav, nav, ... ]
🔹 Storage layout is a first‑order decision. It affects every operator: scanning, filtering, and especially aggregations.

4. Indexing: B‑tree vs. hash index

A: "You have columnar storage. But you still need point lookups for 'order by order_id'. Hash index is O(1). Why not always use hash?"
Name the query that B‑tree handles but hash can't — and explain the storage implication.
A
"Hash index: fast equality. What's missing?"
B
"Range queries: 'WHERE order_date BETWEEN '2026-01-01' AND '2026-03-31''. Hash can't do that efficiently — you'd scan all keys."
A
"So B‑tree. But B‑tree has higher overhead and page splits."
B
"We use B+tree with prefix compression. And for the 'order_id' primary key, we use a clustered index so the row data is stored in leaf pages. That's the key decision: primary key = data order."
B+tree: [root] → [internal] → [leaf pages with sorted keys + row pointers] Range scan: traverse leftmost leaf, then follow sibling pointers.
🔹 Index choice is permanent: it dictates data ordering, storage clustering, and what queries are fast. Later, we'll build on this for joins.

5. Joins: nested‑loop vs. hash join

A: "You have two tables: orders (10M) and users (1M). Nested‑loop is O(n*m). How do you avoid that?"
You can't always build a hash table. What's the constraint that forces a different join strategy?
A
"Hash join: build hash table on smaller table, probe with larger. But what if the smaller table is 10M rows? That's too big for memory."
B
"Then we use sort‑merge join. But that requires both inputs sorted on the join key. If they're not, we have to sort — which is expensive."
A
"So you need a cost‑based optimizer to choose."
B
"Exactly. We estimate cardinalities and choose: hash join if build side < memory threshold, sort‑merge if we have indexes on join key, nested‑loop for small outer."
Cost model: hash join: cost = build_cost(inner) + probe_cost(outer) sort-merge: cost = sort_cost(inner) + sort_cost(outer) + merge_cost
🔹 Join strategy isn't fixed — it's the first real optimizer decision. From now on, everything is cost‑based.

6. Ordering & Top‑K sort

A: "SELECT ... ORDER BY amt LIMIT 10. If you sort the whole table, you're wasting time. What's the better mechanism?"
Don't say 'external sort'. Describe the data structure that gives you the top‑K without full sort.
A
"External merge sort is classic. But for top‑K you don't need full order."
B
"We use a priority queue (min‑heap) of size K. We scan the table, push each row, and if heap exceeds K, pop the smallest. At the end, the heap contains the top‑K."
A
"That's O(n log K). Much better than O(n log n). But what if there's a tie and you need deterministic ordering?"
B
"We include a tie‑breaker column (e.g., order_id) in the heap comparator. That ensures stable ordering, which is critical for paginated IPO allotment reports."
🔹 Top‑K optimization is a cheap win. But it introduces the idea of memory‑aware operators — a theme we'll revisit.

7. Aggregations: group by & hash agg

A: "SELECT amc, SUM(nav) FROM orders GROUP BY amc. You can use a hash table. But what if the number of groups is huge (e.g., 1M groups)?"
What happens when the hash table for groups doesn't fit in memory? How do you handle it without killing performance?
A
"Hash aggregate: build a hash map from group key to accumulator. But with 1M groups, that map may exceed memory."
B
"We use two‑phase aggregation: first, a partial aggregation in each thread (local hash map), then a merge phase. If the total number of groups is still huge, we spill to disk using sort‑based aggregation."
A
"So you're choosing between hash and sort‑based aggregation based on cardinality estimates."
B
"Yes. The optimizer estimates group cardinality from stats. If it's low, hash; if high, sort‑merge aggregation. This is critical for LAMF's multi‑AMC reporting."
🔹 Aggregation forces a spill‑aware design. This spills over into every operator that needs memory (joins, sorts).

8. Transactions: ACID? txn

A: "You have multiple microservices updating orders. Do you support ACID? If yes, show me the mechanism for isolation."
You can't just say 'MVCC'. What exact data structure do you need to implement snapshot isolation?
A
"ACID is expensive. Do you really need it?"
B
"For mutual fund order flow, yes: we can't double‑allot IPO shares. We need serializable isolation for allotment operations."
A
"How do you implement that efficiently?"
B
"We use MVCC with a transaction manager. Each row has a creation and deletion timestamp (or version). Reads see a consistent snapshot; writes create new versions. Conflict detection is done at commit time using a write‑set."
Row version: [ id | amt | creator_txn | deleter_txn ] Snapshot rule: read only versions where creator_txn < snapshot_txn AND (deleter_txn IS NULL OR deleter_txn > snapshot_txn)
🔹 MVCC is the foundation for concurrency and recovery. Every read/write path must consider versions.

9. Concurrency: lock vs. latch conc

A: "MVCC gives you isolation. But you still need to protect internal data structures (e.g., B‑tree pages). What's the granularity?"
Don't say 'lock manager'. Describe the exact problem with page‑level locks for a B‑tree.
A
"You have MVCC for rows. But concurrent inserts into a B‑tree can corrupt it. How do you prevent that?"
B
"We differentiate between locks (transaction‑level, long‑lasting) and latches (mutexes, short‑lived). For B‑tree pages, we use latch‑coupling: acquire latch on parent, then child, release parent."
A
"That's textbook. But what about deadlocks between latches?"
B
"We enforce a strict ordering: always lock left‑to‑right. For page splits, we use a special 'optimistic' latch that avoids deadlocks by restarting the traversal if a split is detected."
🔹 Concurrency control is layered: MVCC for transactions, latches for physical structures. This separation is crucial for performance.

10. Recovery: WAL & checkpoints recovery

A: "The system crashes after a commit. How do you guarantee durability without fsync on every write?"
You can't fsync every commit. What's the compromise that still gives durability?
A
"Write‑ahead log (WAL) — write the log before the data page. But fsync on every commit is slow."
B
"We group commits: accumulate several transactions in the WAL buffer, then fsync once. The recovery process replays the WAL from the last checkpoint."
A
"What if the WAL itself is corrupted?"
B
"We use a checksum on each WAL record. And we keep a mirrored WAL for critical NSDL demat operations. Recovery is a two‑phase: redo all committed, undo all aborted."
🔹 WAL and checkpoints are the bedrock of durability. Every write path must go through the log.

11. Distributed: sharding & replication dist

A: "80 microservices mean you can't have a single node. How do you shard orders without breaking joins?"
Sharding by order_id is easy. But what about a join with users table that is sharded differently?
A
"Shard orders by order_id. But joins with users (sharded by user_id) become cross‑shard."
B
"We use a two‑level strategy: orders sharded by order_id, but we also replicate a subset of users (by frequent join keys) to each shard. For ad‑hoc cross‑shard joins, we use a distributed hash join with a shuffle stage."
A
"That's expensive. What's the fallback?"
B
"We push down filters and aggregations to each shard, then merge results. For joins that can't be pushed, we use a broadcast join if one side is small."
🔹 Distributed execution is the ultimate trade‑off: you trade network for scale. Every operator must be partition‑aware.

12. The hardest question: what would you redesign? rethink

A: "You've built this engine for FinTech, with MVCC, WAL, sharding, cost‑based optimization. If you could start over, what's the one thing you'd do differently?"
No reveal. Sit with it. What's the most painful trade‑off you made? (Then click to see my answer.)
B
"I'd decouple storage from compute earlier. Our columnar format is great for analytics, but it makes point updates painful. I'd adopt a 'delta‑store + base‑store' model like LSM trees, so we get fast writes and still have columnar scans."
A
"That's a real trade‑off. So you're saying the storage layer is the most fundamental decision."
B
"Yes. Everything else — indexing, joins, recovery — sits on top. If you get storage wrong, you're fighting the hardware."
🔹 The final insight: every component is a response to a previous problem. The engine is not a checklist — it's a chain of decisions.
